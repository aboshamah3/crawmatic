# Phase 0 Research: Database Foundation

All decisions below were resolved from `PROJECT_SPEC.md` (§4, §6, §19, §21, §22, §32), the constitution (Principles II, VII, VIII), and the SPEC-01 skeleton. Each was verified against a live SQLAlchemy/`uuid6` install in this environment where noted. No open `NEEDS CLARIFICATION` remain.

---

## D1 — UUIDv7 generation library

**Decision**: Use the pure-Python **`uuid6`** package (`uuid6.uuid7()`), pinned `uuid6>=2025.0.1,<2026`. Wrap it in `app_shared/ids.py` as `new_uuid7() -> uuid.UUID` and use it as the SQLAlchemy PK `default` (Python-side callable). Store in a Postgres native `UUID` column (SQLAlchemy `sqlalchemy.Uuid` / `postgresql.UUID(as_uuid=True)`).

**Rationale**:
- Verified here: `uuid6.uuid7()` returns an object that **is** a `uuid.UUID` subclass (`isinstance(x, uuid.UUID) == True`), reports `.version == 7`, and is monotonically time-ordered (`str(a) < str(b)` for `a` generated before `b`). So it drops straight into SQLAlchemy's `Uuid` type and into psycopg's UUID adaptation with no custom marshalling.
- Pure Python, MIT-licensed, zero native/build dependencies — no wheel/ABI risk in the migration-job or app images, unlike a Rust-backed alternative.
- App-generated (not DB-generated) satisfies §21 "can be generated in the app" and keeps the ID available before flush (needed for FK wiring and returning IDs from writes without a round-trip).
- Python 3.13 has **no** stdlib `uuid.uuid7()` (that lands in 3.14), so a library is required; `requires-python = ">=3.13,<3.14"` pins us below the stdlib option.

**Alternatives considered**:
- **`uuid-utils`** (Rust/PyO3, faster): rejected — native build/ABI surface for negligible perf gain; UUID generation is not a hot path (a scrape refresh generates thousands, not millions, of IDs, and generation is nanoseconds).
- **Postgres-side `gen_uuid_v7()` / DB default**: rejected — §21 mandates app-generation; also needs an extension or a hand-written SQL function and hides the ID until after insert.
- **Hand-rolled generator**: rejected — reinventing a spec-compliant RFC 9562 UUIDv7 (timestamp + monotonic counter + randomness) is unjustified risk when a small, vetted, pure-Python lib exists.

**Storage note**: column type is `Uuid(as_uuid=True)` → Postgres `UUID`. Accepted trade-off (§21): UUIDv7 embeds creation time, so public IDs disclose row-creation time — documented and acceptable for this product.

## D2 — Deterministic naming convention (all constrained columns)

**Decision**: A single shared `MetaData(naming_convention=...)` on the declarative `Base`, using the `column_0_N_name` token (all columns, underscore-joined) for `ix`/`uq`/`fk`:

```python
NAMING_CONVENTION = {
    "ix":  "ix_%(table_name)s_%(column_0_N_name)s",
    "uq":  "uq_%(table_name)s_%(column_0_N_name)s",
    "ck":  "ck_%(table_name)s_%(constraint_name)s",
    "fk":  "fk_%(table_name)s_%(column_0_N_name)s_%(referred_table_name)s",
    "pk":  "pk_%(table_name)s",
}
```

**Verification (run in this environment)**: a `products` table with `UniqueConstraint("workspace_id", "external_id")` and `UniqueConstraint("workspace_id", "sku")` produced exactly:
- `uq_products_workspace_id_external_id`
- `uq_products_workspace_id_sku`

Two **distinct** deterministic names — the collision the default `%(column_0_name)s` would cause (`uq_products_workspace_id` twice) is eliminated. This is FR-002 / SC-003 satisfied.

**Rationale**:
- `%(column_0_N_name)s` is SQLAlchemy's built-in token that expands to *every* column in the constraint joined by `_` (vs. `%(column_0_name)s` = first column only, vs. `%(column_0N_name)s` = all columns with **no** separator, which is harder to read). It is deterministic and autogenerate-stable, so Alembic autogenerated migrations reproduce the same names (FR-014).
- `ck` uses `%(constraint_name)s`: check constraints have no reliably-introspectable column set, so the convention **requires developers to give each `CheckConstraint` an explicit name** — an intentional forcing function (an unnamed check constraint fails to render, surfacing the omission at model-definition time).

**Known limitation & mitigation**: Postgres identifiers truncate at 63 bytes; a convention that concatenates many long column names could exceed that and re-collide after truncation. For the tables in this schema (§22 uniques are 2–3 short columns) names stay well under 63. Mitigation path if a future table approaches the limit: replace the token with a custom naming *function* that truncates deterministically and appends a short hash of the full column list. Not needed now; documented so it is a conscious future choice, not a surprise.

**Alternatives considered**:
- Default SQLAlchemy convention (`column_0_name`): rejected — collides on the exact two-uniques case §22 calls out.
- Custom Python naming function with hashing: rejected as premature — the built-in token disambiguates within the 63-char budget for every current/planned table; adding a hash now trades readability for unneeded robustness.

## D3 — Forbidding naive timestamps at the base level

**Decision**: A `TimestampMixin` provides `created_at` and `updated_at` as `mapped_column(DateTime(timezone=True), ...)` → Postgres `TIMESTAMPTZ`, defaulted with a UTC-aware callable (`lambda: datetime.now(timezone.utc)`) and `onupdate` for `updated_at`. Enforce naive-rejection two ways:

1. **Type-level guard**: a small `TZDateTime(TypeDecorator)` (over `DateTime(timezone=True)`) whose `process_bind_param` raises `ValueError` if a `datetime` is passed with `tzinfo is None`. This makes it impossible to *store* a naive value even if a caller supplies one. The mixin's columns use this type.
2. **Schema-level guarantee**: `DateTime(timezone=True)` renders `TIMESTAMPTZ`, so the *column* is always timezone-aware in Postgres; there is no naive-timestamp column type reachable through the base.

Developers are directed to derive timestamp columns from the mixin / `TZDateTime`; a naive `DateTime()` (no `timezone=True`) is a lint/review target but the type guard is the hard, testable stop.

**Rationale**: §22 "naive timestamp columns are forbidden — enforce at the base-model level so it cannot be forgotten per table." A bind-param guard is DB-independent and unit-testable here (assign a naive `datetime` → `ValueError`); the `TIMESTAMPTZ` render is verified offline via Alembic `--sql`.

**Alternatives considered**:
- SQLAlchemy event listener on `before_insert`/`before_update` per model: rejected — per-model wiring is exactly the "forgotten per table" failure mode; a shared type/mixin is inherited automatically.
- Relying on `DateTime(timezone=True)` alone: insufficient — Postgres `TIMESTAMPTZ` will still *accept* a value the driver localizes from a naive datetime; the app-side guard is what refuses naive input at the boundary.

## D4 — Money SQLAlchemy TypeDecorator

**Decision**: `app_shared/money.py` defines `Money(TypeDecorator)` with `impl = Numeric(precision=18, scale=4, asdecimal=True)` and `cache_ok = True`. Validation in `process_bind_param(value, dialect)`:

1. `None` passes through (nullable columns allowed).
2. Reject `float`/`int`-as-float and `str`? — accept `Decimal` and `int`; **reject `float`** outright (floats can't represent money exactly — §19 "never float"). Callers must pass `Decimal` (or `int`). A `str` is accepted by converting via `Decimal(str_value)` (exact) — optional convenience; `float` is the hard reject.
3. Coerce accepted input to `Decimal`; reject non-finite: `if not value.is_finite(): raise ValueError` (covers `NaN`, `Infinity`, `-Infinity`) — §19 "reject NaN and Infinity at the type boundary."
4. Reject over-scale: if the value has more than 4 fractional digits (`-value.as_tuple().exponent > 4`), raise `ValueError` — §19 "values with more decimal places than the column scale are rejected, not silently rounded."

`process_result_value` returns the `Decimal` unchanged (SQLAlchemy `Numeric(asdecimal=True)` already yields `Decimal`), guaranteeing round-trips never surface a float.

**Rationale**: Puts the §19/Principle-VII guarantees at the *type boundary* so every money column inherits them; fully DB-independent and unit-testable here (assign `Decimal("NaN")`, `Decimal("Infinity")`, `Decimal("1.23456")`, `1.1` float → each raises; `Decimal("19.9900")` and `Decimal("0.0001")` round-trip exactly). `cache_ok = True` lets SQLAlchemy cache statements using the type.

**Alternatives considered**:
- Plain `Numeric(18,4)` columns: rejected — Postgres `NUMERIC` stores `NaN` and silently accepts/rounds; §19 explicitly requires app-side refusal.
- A pydantic-only validator at the schema layer: rejected/insufficient — schemas are a later spec and don't cover direct ORM writes; the guarantee must live on the DB type so it can't be bypassed.
- Storing minor units as integers (cents): rejected — §19 mandates `NUMERIC(18,4)` `Decimal`, and 4-dp scale needs sub-cent precision anyway.

## D5 — Workspace-scoped base + RLS policy DDL helper

**Decision**:
- `WorkspaceScopedBase` (a mixin usable with the declarative `Base`) adds `workspace_id: Mapped[uuid.UUID] = mapped_column(Uuid, nullable=False, index=True)`. No FK to `workspaces` here (that table is SPEC-03) — it is a plain indexed UUID column; SPEC-03 adds the FK when `workspaces` exists.
- `app_shared/models/rls.py` exposes `emit_rls_policy(table_name, *, workspace_column="workspace_id", policy_name=None) -> tuple[str, ...]` returning the DDL statements (as text, for use inside an Alembic migration via `op.execute(...)`):
  1. `ALTER TABLE {t} ENABLE ROW LEVEL SECURITY;`
  2. `ALTER TABLE {t} FORCE ROW LEVEL SECURITY;` (so even the table owner is subject to the policy)
  3. `CREATE POLICY {policy} ON {t} USING ({col} = NULLIF(current_setting('app.workspace_id', true), '')::uuid);`  ← NULLIF(…, '') → absent AND empty context both map to NULL → zero rows (never raises on empty)

**Fail-closed semantics**: `current_setting('app.workspace_id', true)` returns `NULL` when the GUC is unset or empty (the `true` = "missing_ok" argument suppresses the error). `NULL = anything` is `NULL` (not true), so the `USING` predicate matches **zero rows** when no workspace context is set — isolation fails closed (§32, spec Edge Case). Application code sets context per transaction with `SET LOCAL app.workspace_id = '<uuid>'` (transaction-scoped — the only form safe under PgBouncer transaction pooling).

**Rationale**: §32 mandates RLS on every workspace-owned table, enabled in the same migration that creates the table, fail-closed, `SET LOCAL`-based. SPEC-02 delivers the *machinery* (mixin + DDL emitter) and validates it statically (assert the rendered DDL string contains `ENABLE ROW LEVEL SECURITY`, `FORCE`, and the fail-closed `NULLIF(current_setting('app.workspace_id', true), '')::uuid` predicate). No real workspace-owned table is created here; first concrete use is SPEC-03. The smoke/demo table is **not** workspace-owned (keeps SPEC-02 strictly free of real isolation surface) — the RLS helper is exercised by a rendered-DDL unit test, and optionally by applying it to the demo table only inside a live-DB integration test marked for a PG host.

**Alternatives considered**:
- Emitting RLS via a SQLAlchemy DDL event on the mapper: rejected — §32 wants RLS in the *migration* that creates the table (explicit, reviewable, versioned), not implicitly at metadata-create time.
- A permissive policy defaulting to "all rows when unset": rejected — violates fail-closed; an unset context must yield zero rows, never all rows.

## D6 — Alembic layout & migration-job direct-URL wiring

**Decision**:
- `alembic.ini` at repo root; `alembic/env.py` sets `target_metadata = app_shared.models.Base.metadata` (single shared metadata → autogenerate honors the naming convention, FR-014), enables `compare_type=True`, and runs online migrations using an engine built from **`MIGRATION_DATABASE_URL`** (direct-to-Postgres), falling back to a CLI `-x db_url=` override. `env.py` builds its own engine (does *not* reuse `app_shared.database.get_engine()`, which targets the pooler) so migrations never go through PgBouncer.
- **Migration job**: new `apps/migrate/Dockerfile` (python base, installs from the root uv lockfile so migrations run under the exact pinned versions — §5) with entrypoint `alembic upgrade head`. A compose **one-shot** `migrate` service (`restart: "no"`, `depends_on: postgres: service_healthy`, `command: alembic upgrade head`) connects to `postgres:5432` directly via `MIGRATION_DATABASE_URL`. App services (`api`/`scheduler`/`worker`) never run migrations at startup (FR-011, §6) — they only `depends_on` pgbouncer. **Cannot be executed here** (no Docker daemon) — authored and marked; run on a PG-capable host.
- **Direct-URL source**: `MIGRATION_DATABASE_URL=postgresql+psycopg://crawmatic:crawmatic@postgres:5432/crawmatic` in `.env.example`, distinct from `DATABASE_URL` (→ `pgbouncer:6432`). Rationale for direct: session-scoped advisory locks (Alembic's version-table locking) and `CREATE INDEX CONCURRENTLY` are unsafe/broken through transaction pooling (§4/§6).
- **Single linear history / CI guard**: `scripts/check_single_head.sh` runs `alembic heads` and fails if it reports != 1 head (FR-012, SC-006). Runs in CI; DB-independent (reads migration files, no connection).

**Rationale**: Matches §6's "single exception is the one-shot migration job: it connects directly to Postgres and is never run automatically by an application service at startup." `compare_type=True` keeps autogenerate honest about `TIMESTAMPTZ` vs `TIMESTAMP` and `NUMERIC` scale. Installing from the root lockfile pins migration-time dependency versions (§5).

**Alternatives considered**:
- Running migrations from the API container on boot: rejected — §6/FR-011 forbid it (unsafe under pooling, non-deterministic ordering).
- Pointing `env.py` at the pooler: rejected — advisory locks + concurrent index builds break under transaction pooling.
- Async Alembic: rejected — stack is sync SQLAlchemy over psycopg 3 (spec Clarifications).

## D7 — Demonstration / smoke strategy (no live Postgres here)

**Decision**: A tiny **demonstration model** in the test/support surface plus a **first real Alembic migration** that creates one small permanent bookkeeping/demo table, exercising the full machinery:

- **Demo model** `_SmokeFoundation` (test-support, e.g. defined in `tests/support/` or the migration module — deliberately *not* a domain table): UUIDv7 PK from the base, `TimestampMixin` (tz-aware `created_at`/`updated_at`), a `Money` column, a string-backed enum column, and **two multi-column unique constraints sharing a first column** (`uq(group_key, code_a)` and `uq(group_key, code_b)`) so the naming disambiguation is exercised end-to-end.
- **First migration** `<rev>_smoke_foundation.py`: `create_table(...)` for the demo table with the generated constraint names; it is the head revision. It demonstrates (in a comment/optional `op.execute`) how a *future* workspace-owned table would call `emit_rls_policy(...)` — but since the demo table is not workspace-owned, RLS is only rendered in the unit test, not applied to a real isolation surface.

**What is verifiable HERE (DB-independent, unit tests)**:
- Naming convention → distinct names for the two uniques (metadata introspection). ✅ verified.
- UUIDv7 default returns a v7, time-ordered, stdlib `UUID`. ✅ verified.
- Money type rejects NaN/Infinity/over-scale/float and round-trips in-scale `Decimal`.
- Naive-datetime guard raises.
- `emit_rls_policy()` renders the fail-closed DDL string.
- **Offline migration render**: `alembic upgrade head --sql` produces SQL text (no DB connection) — assert it contains `CREATE TABLE`, `TIMESTAMPTZ`, `NUMERIC(18, 4)`, `uuid`, and the two distinct unique names. This proves the migration is well-formed and that `TIMESTAMPTZ`/`NUMERIC(18,4)` render correctly without a live server.
- Single-head guard (`alembic heads` == 1).

**What is MARKED for a Postgres host (live-DB integration tests, skipped here)**:
- `alembic upgrade head` against a real database creates the demo table (SC-001, AS-US1).
- `check_connection()` executes `SELECT 1` (FR-015, SC-007).
- The migration job connects directly, not via the pooler (AS-US1-2).

**Rationale**: Gives a genuine, reviewable first migration and full DB-independent proof of every convention now, while cleanly deferring the handful of items that truly need a server — exactly the split the spec Assumptions and the constraint "no Docker daemon in build env" require. Keeping the smoke table tiny and non-domain honors strict scope (no real domain tables).

**Alternatives considered**:
- Offline-rendered SQL only, no permanent table: rejected — a real head migration is needed anyway to bootstrap the history and to give the live-host acceptance test something to apply; a permanent tiny demo table is the least-effort way to make SC-001 executable on a host.
- A throwaway migration later downgraded/deleted: rejected — churns the linear history; a small stable demo table is simpler and harmless.

## Summary of pinned choices

| Concern | Choice |
|---|---|
| UUIDv7 | `uuid6>=2025.0.1,<2026`, `uuid6.uuid7()` wrapped as `new_uuid7()`, stored as Postgres `UUID` |
| Naming | `MetaData(naming_convention=...)` with `column_0_N_name` for ix/uq/fk; explicit-name `ck`; verified to disambiguate the two shared-first-column uniques |
| Naive TS | `TZDateTime` TypeDecorator (bind-param guard) + `DateTime(timezone=True)`/`TIMESTAMPTZ` on `TimestampMixin` |
| Money | `Money(TypeDecorator)` over `Numeric(18,4, asdecimal=True)`; reject float/NaN/Infinity/over-scale; return `Decimal` |
| RLS | `WorkspaceScopedBase` (adds `workspace_id`) + `emit_rls_policy()` → ENABLE+FORCE+fail-closed policy DDL; first real use SPEC-03 |
| Alembic | root `alembic.ini`/`env.py`; `target_metadata = Base.metadata`; `compare_type`; direct `MIGRATION_DATABASE_URL`; one-shot compose `migrate` service; `check_single_head.sh` CI guard |
| Config | add optional `MIGRATION_DATABASE_URL` (direct); apps keep `DATABASE_URL` (pooler) |
| Demo | permanent tiny non-domain smoke table + first migration; DB-independent parts unit-tested; live parts marked for a PG host |
