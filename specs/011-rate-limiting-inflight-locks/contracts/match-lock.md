# Contract: In-Flight Match Lock (fencing token)

**Module**: `libs/shared/app_shared/limiter/locks.py` (pure Redis logic; `redis.Redis`-shaped
client param; stdlib `secrets` only; no Scrapy/Twisted). Covers FR-010..FR-014, FR-016; US2;
SC-002, SC-004.

---

## Key & value
- **Key**: `lock:scrape:{workspace_id}:{match_id}` (FR-010; built by `keys.match_lock_key`).
- **Value**: a unique **fencing token** — `secrets.token_hex(16)` (128-bit), generated by the
  acquirer and threaded to the releaser.

## API

```python
def new_fencing_token() -> str: ...   # secrets.token_hex(16)

def acquire_match_lock(redis, *, key: str, token: str, ttl_seconds: int) -> bool:
    """SET key token NX PX ttl_seconds. True = acquired (we own it); False = already held."""

def release_match_lock(redis, *, key: str, token: str) -> bool:
    """Lua compare-and-delete: DEL only if GET==token. True = we released our own lock."""
```

- **acquire** = single `SET key token NX PX ttl` round-trip (FR-010). `False` (NX fails) ⇒ the
  match is already being scraped ⇒ caller marks the target `SKIPPED` / `LOCKED_ALREADY_RUNNING`
  and does **no** fetch (FR-014, US2 AS1).
- **acquire Redis error ⇒ return `False` (fail-closed — treat "cannot confirm ownership" as
  do-not-fetch, FR-023, D3).**
- **release** = atomic Lua:
  ```lua
  if redis.call('GET', KEYS[1]) == ARGV[1] then return redis.call('DEL', KEYS[1]) else return 0 end
  ```
  Deletes only when the stored token still matches (FR-012). An expired-then-reacquired lock
  owned by a new holder is **preserved** — the slow prior owner's release is a no-op (US2 AS3).
  Release Redis error ⇒ logged + swallowed (TTL reclaims; never crash persistence — D3).

## TTL (mode-sized — FR-013)
- HTTP mode: `Settings.MATCH_LOCK_HTTP_TTL_SECONDS` (~600s).
- Browser mode (`PLAYWRIGHT_PROXY`, reserved for SPEC-14): `MATCH_LOCK_BROWSER_TTL_SECONDS` (~1800s).
- **Invariant**: chosen TTL ≫ worst-case in-spider wait
  `REQUEUE_MAX_ATTEMPTS × (wait_hint + RATE_LIMIT_JITTER_MAX_SECONDS)` so a still-working owner
  never loses its lock mid-fetch (US2 AS4) — asserted in unit tests.

## Ownership lifecycle (FR-011, §13)
1. **Spider** generates a fencing token and `acquire_match_lock` **immediately before fetch**
   (in `start()`/`errback()`, after the limiter grant — see `spider-integration.md`).
2. Token is stamped on `request.meta["match_lock_token"]` → carried onto the emitted
   `ScrapeResult`.
3. **Spider's persistence pipeline** calls `release_match_lock` with that token **after** the
   observation/attempt write completes, in the same off-reactor flush (`pipelines.py`).
4. The worker never holds a per-match lock across the dispatch boundary (§13).
5. `price_analysis` runs after release and needs no lock (FR-016; idempotent per variant).

## Job-level dedup (verify only — FR-015)
`unique(scrape_job_id, match_id)` on `scrape_job_targets` (SPEC-08,
`uq_scrape_job_targets_scrape_job_id_match_id`) prevents a duplicate target within one job
**before** any lock is attempted (US2 AS5). SPEC-11 **verifies** this constraint exists (a
skip-clean migration/integration test) — it does **not** re-add it.

## Invariants (tested — SC-002)
- Two concurrent `acquire_match_lock` on one key ⇒ exactly one `True`.
- `release_match_lock(token=other)` on a lock held by `token=mine` ⇒ `0` (no delete).
- After the owner releases (or TTL lapses), the key is re-acquirable.
